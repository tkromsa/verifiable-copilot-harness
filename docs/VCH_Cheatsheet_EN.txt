Verifiable Copilot Harness - Quick Use Guide v6.15.0

WHAT IT IS
A governed set of 43 skills and verification rules for Copilot work. It supports lightweight non-project work, controlled project creation and self-contained project workbooks without requiring every VCH file in every session. From v6.13.0 it also carries an embedded, out-of-bootstrap behavioral regression corpus (__ROUTING_ORACLE).

THE CORE IDEA
Normal operation uses one active workbook.
- HARNESS MODE
File: VCH_HarnessCore.xlsx
Use: Everyday non-project work, analysis, debugging, planning and skill-driven tasks.
Result: No project workbook is created unless you later add the template and explicitly request PROJECT-FORK.
- TEMPLATE MODE
File: VCH_ProjectTemplate.xlsx
Use: Start a project directly.
Result: PROJECT-FORK creates revision 001. The template remains immutable.
- PROJECT-CREATION MODE
Files: VCH_HarnessCore.xlsx + VCH_ProjectTemplate.xlsx
Use: Convert useful non-project work into a governed project.
Result: The template is the physical fork source. Compatible duplicate behavior is validated but read only once.
- PROJECT MODE
File: Current <Project_ID>_<Safe_Project_Name>_vNNN.xlsx
Use: Continue an existing project.
Result: The project workbook is self-contained. Core and template are not required for normal continuation.
- MIGRATION MODE
Files: Old project workbook + current VCH_ProjectTemplate.xlsx
Use: Move verified project content to the current schema.
Result: The old project remains read-only. A new project line is created from the current template.

FILES
VCH_HarnessCore.xlsx
- Complete behavior-only kernel: rules, 43 skills, oracle, routing oracle, state schema, lists, ADR and glossary.
- Never use as a project state anchor.
- Never write project delivery content into it.
VCH_ProjectTemplate.xlsx
- Immutable standalone project starter.
- Contains the complete behavior snapshot, routing oracle and project delivery sheets.
- Physical source for PROJECT-FORK.
Current project workbook
- Revision 001 or higher.
- Contains the behavior snapshot, project identity, state and delivery content.
- Default active workbook after the fork.
copilotstart.txt
- Runtime contract for mode selection, bootstrap, persistence, evidence and the routing oracle.
copilot_custom_instructions.txt
- Paste once into Copilot Custom Instructions.
- Holds compact always-on invariants and workbook mode selection.
harness_lint.ps1
- Integrity validator for harness and template changes.
- Not required for normal project delivery unless the harness or template itself is changed.

DAILY START
Non-project work:
- Attach VCH_HarnessCore.xlsx.
- Type: load vch
- Use a skill or ask normally.
Start a project directly:
- Attach VCH_ProjectTemplate.xlsx.
- Type: load vch
- Type [skill: PROJECT-FORK] and provide Project ID, Project Name, Owner and Main Goal.
Convert current work into a project:
- Keep VCH_HarnessCore.xlsx available and attach VCH_ProjectTemplate.xlsx.
- Type [skill: PROJECT-FORK] with complete project metadata.
- After verified revision 001 is created, use only the new project workbook for normal continuation.
Continue a project:
- Attach only the current verified project workbook and files needed for the current task.
- Type: load vch
- Type: guide project

SOURCE SELECTION RULES
- Explicit binary attachment beats Work IQ search results for editing and persistence.
- Current project workbook beats template for project continuation.
- Template beats core as the physical fork source.
- Older revisions are lineage evidence only when requested.
- Multiple possible current revisions return AMBIGUOUS_LINEAGE. Filename recency alone is not proof.

BOOTSTRAP OUTPUT
The read-only bootstrap reports:
- WORKBOOK_MODE
- SELECTED_BEHAVIOR_SOURCE
- Harness version
- Skill count
- File role
- Goal
- Revision and lineage
- Resume_From
- Host_Mode
- Capability manifest
- Persistence_Mode
The bootstrap read-set is Project.Rules, 00_Landing, __STATE, 00_Skills, __TEST_ORACLE, __ADR and __GLOSSARY. __ROUTING_ORACLE is NOT read on load.

PRIMARY COMMANDS
load vch
- LOAD-ALL and BOOTSTRAP-CHECK.
- Always read-only.
guide project / what next
- PROJECT-GUIDE selects one specialist skill and checks phase gates.
new project / project fork
- PROJECT-FORK creates revision 001 from the template.
save revision / commit revision / persist changes
- REVISION-COMMIT creates the next verified physical revision where supported.
write plan
- WRITING-PLANS creates steps with VERIFY evidence and confidence.
debug this
- SYSTEMATIC-DEBUGGING finds root cause before proposing a fix.
analyze data
- DATA-ANALYSIS produces quantified findings and identifies data-quality gaps.
review code
- CODE-REVIEW reports findings without rewriting the source.
security review / security audit
- SECURITY-REVIEW performs a structured security assessment.
verify fix
- VERIFICATION-CHECKLIST requires real evidence. No error is not proof.
compound / record solution
- COMPOUND records a verified and sanitized reusable solution.
update context
- UPDATE-CONTEXT saves verified state only.
scrub this / scrub output
- SCRUB removes sensitive values before output crosses a trust boundary.
focus mode / adhd mode
- FOCUS-MODE activates action-first output.
- Deactivate with: focus mode off
status / where are we / current status
- STATUS returns only the Status Card (Mode, Resume, Gate, Verified, Next). Read-only.
what can you do / available skills
- CAPABILITY-DISCOVERY lists only skills usable right now (Allowed_Modes column, gates, capability). Never the full catalog.
run routing oracle
- Behavioral regression run of __ROUTING_ORACLE (not a formal trigger; explicit request only).

ROUTING ORACLE (behavioral regression, out-of-bootstrap)
- Lives in the __ROUTING_ORACLE sheet inside the behavior source, next to __TEST_ORACLE, but is never read on load and never runs automatically.
- Two layers, one sheet, one Layer column:
  L2 (routing correctness): exact triggers, alt atoms, explicit [skill: ID], prefix pasti (script review vs review), precedence tie-break, guards (WRITE_BLOCKED), ambiguity, focus mode, negative and chain checks.
  L3 (adversarial / safe failure): write-trap, template-trap, SCRUB-completeness, probe-trap, precopy precondition, enum injection, capability inheritance, ambiguous lineage, version mismatch, self-report, character policy, duplicate catalog.
- How to run: on explicit request or before promoting a catalogue change, feed each Input_Utterance in an isolated context and compare the produced Skill_ID and verdict to the frozen Expected_Skill_ID and Expected_Verdict. Optionally run the same corpus on two models (A/B) against identical expected values; a disagreement flags a routing ambiguity.
- Scoring: PASS when skill and verdict both match; PARTIAL when only the skill matches; FAIL on the wrong skill. A FAIL on any Critical=YES GUARD or L3 row is release-blocking.
- What the lint does vs does not: harness_lint checks only structure (every Expected_Skill_ID is NONE or an existing Skill_ID; every Test_ID is unique). It never asserts the routing decision. The routing decision is a behavioral test executed by the model.

FROZEN ORACLE (structural)
- __TEST_ORACLE remains the structural gate: SKILLCOUNT=43, version agreement, trigger duplicates and prefix collisions, non-ASCII, chains, enums, state, probe cell, mode, plus three ROUTING structural rows (present, unresolved skill refs, duplicate Test_ID).

PERSISTENCE MODES
READ_ONLY
- Default when persistence is not requested or capability is not proven.
ARTIFACT_COPY_ON_WRITE
- Chat creates the next physical revision, writes, exports, reopens and verifies it.
PRECOPIED_ARTIFACT
- The user creates the exact next revision with Save a Copy; Copilot verifies and activates it.
EXCEL_NATIVE_VERSIONING
- OPEN_WORKBOOK logical checkpoint after direct edit is proven and persistence is explicit.
WRITE_BLOCKED
- Persistence was requested but the required capability or prerequisite was not proven.

NON-NEGOTIABLE RULES
- Bootstrap is always read-only and never reads __ROUTING_ORACLE.
- Never edit HarnessCore or ProjectTemplate for project delivery.
- Filename alone is not lineage evidence.
- UNKNOWN is not PASS.
- Stored capability is never inherited as current runtime capability.
- Output crosses the trust boundary only after SCRUB.
- PROBE_CELL is resolved by name, not a fixed address.

MIGRATION
- Attach the old project workbook and current VCH_ProjectTemplate.xlsx.
- Type: load vch
- Expect VERSION_MISMATCH when versions differ; this is read-only.
- Fork from the current template with the same verified project metadata.
- Port only verified project content by field or column identity.
- Reopen, verify, set Resume_From and archive the old project line.

DO NOT TRANSFER DURING MIGRATION
- Runtime capability state.
- Unverified attempts as verified work.
- Old oracle or Lists over the current template.
- Ambiguous lineage.

COMMON STATUS VALUES
WRITE_BLOCKED
- Persistence is not proven. Continue read-only or provide a valid persistence path.
VERSION_MISMATCH
- Loaded workbook uses an older Template_Version. Perform controlled migration.
SCHEMA_MISMATCH
- Required structure or critical behavior identity differs. Keep the transition read-only.
AMBIGUOUS_LINEAGE
- More than one candidate current revision exists or parentage is unclear.
BLOCKED
- A phase prerequisite is missing.
NOT_TESTED
- No verification evidence exists yet. It is neither PASS nor automatic failure.
PRECONDITION_FAILED
- An input condition is not met, for example a precopied target is not SourceRevision+1.

FAQ
Do I always attach HarnessCore?
No. Attach HarnessCore for non-project work. Use the template to start a project. After revision 001 exists, use the current project workbook for normal continuation.
Can the template be used without HarnessCore?
Yes. The template contains the complete behavior catalog, routing oracle and delivery schema. It is a standalone project starter.
Why does the project workbook contain the skills and rules?
It preserves a self-contained behavior snapshot for auditability, recovery and controlled migration.
What happens if Core and Template are both present?
This is PROJECT-CREATION MODE. The template is the physical fork source. Compatible duplicate behavior is checked but not concatenated.
Does the routing oracle slow down load vch?
No. __ROUTING_ORACLE is out of the bootstrap read-set. It is read only when you explicitly run it or before promoting a catalogue change.
Is the routing oracle a one-off test?
No. It is a frozen regression corpus. Re-run it whenever the catalogue changes (new skill, changed trigger or precedence) to catch routing drift.
What if Copilot finds older revisions through Work IQ?
An explicitly attached current workbook remains the preferred editable artifact. Older revisions are used only when lineage, migration or comparison requires them.
Copilot forgot the harness. What now?
Attach the correct active workbook for the intended mode and type: load vch
Can I add a skill?
Yes, as a governed harness change: add the 00_Skills row, update SKILLCOUNT and documentation, add an ADR, add matching __ROUTING_ORACLE rows, run harness_lint.ps1 and a routing run, and promote only after deterministic checks and the routing run pass.

GOLDEN RULE
Use one active workbook in normal operation. Use multiple workbooks only at a controlled transition boundary.
RELEASE GATES
- Run harness_lint.ps1 against both workbooks.
- Run all __ROUTING_ORACLE rows in isolated contexts; apply Fixture_Context and Candidate_Skill_Set for synthetic tie tests.
- Structural PASS never implies routing PASS. Any Critical=YES mismatch blocks catalogue promotion.
